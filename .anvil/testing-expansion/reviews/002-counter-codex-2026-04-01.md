## What Works

The plan still has the right backbone. The sequencing is mostly sound: schema tightening before contract validation, shared harness before cross-layer tests, and mutation ratchet after broader coverage exists. The citations to existing patterns are generally real and useful: temp-dir SQLite tests, `server.start(0)`, the SSE reader helper, and the visual regression conventions are all grounded in the repo.

The best decision remains direct webhook injection. Posting signed payloads to `/webhooks/linear` is the shortest path through the real trust boundary, and it avoids inventing a fake Linear server that the codebase does not need.

## Settled Points

- `C2` `AGREE` — the denominator is wrong, and Claude was directionally right to call it out. The current OpenAPI spec generated by `src/http/openapi.ts` exposes 20 paths / 23 operations, while the runtime route surface registered across `src/http/routes.ts`, `src/config/api.ts`, `src/secrets/api.ts`, `src/setup/api.ts`, `src/prompt/api.ts`, and `src/audit/api.ts` is 55 method/path combinations. "27 API endpoints" is inaccurate either way.
- `C4` `AGREE` — `src/setup/validate.ts` does not exist.
- `C5` `AGREE` — `RisolutoEventMap` defines 13 channels, not 12.
- `C7` `AGREE` — the same-port restart design has a real flake path. `HttpServer.start()` hard-fails on `EADDRINUSE`, so `stop()` then immediate `start(capturedPort)` needs a retry strategy or a different reconnection design.
- `C8` `AGREE` — there is no `.github/workflows/nightly.yml`; the only relevant workflow is `.github/workflows/ci.yml`, and its schedule is weekly (`0 6 * * 1`), not nightly.
- `C13` `AGREE` — the proposed file-specific integration scripts are rename-fragile.
- `C15` `AGREE` — the contract-test scope is materially underspecified relative to the runtime API surface. Setup, template, and audit endpoints are not in the current OpenAPI spec, and the spec also omits `/api/v1/models`, `/api/v1/events`, `/api/docs`, and webhook routes.
- `C18` `AGREE` — `openDatabase()` is schema bootstrap via `CREATE TABLE IF NOT EXISTS`, not a SQL migration runner. The real migration-like behavior in the SQLite runtime is JSONL import via `migrateFromJsonl()` in `src/persistence/sqlite/runtime.ts`.

## Contested Points

- `C1` `PARTIALLY AGREE` — raw bytes do matter, but "use `JSON.stringify(payload)`" is not inherently wrong if that exact serialized string is also used as the request body. The existing webhook tests already follow this pattern: sign a string, then use the same bytes as `rawBody`. The plan should say "sign the exact serialized request body" instead of implying any `JSON.stringify(...)`-based test is invalid.
- `C3` `PARTIALLY AGREE` — the plan path is wrong, but Claude's proposed replacement is also wrong. There is no `src/github/github-pr-client.ts`, but there is a real `src/git/github-pr-client.ts`. `src/github/issues-client.ts` is a different client.
- `C6` and `C16` `PARTIALLY AGREE` — the harness is underspecified, especially for webhook cases, but "you must build a partial real Orchestrator" overstates it. Existing tests show thin `OrchestratorPort` stubs are acceptable for much of the HTTP surface. What the plan actually needs is a two-tier design: a base server harness with stub-friendly orchestrator methods, plus opt-in real wiring for event bus, webhook inbox, and secret-backed webhook deps.
- `C9` `PARTIALLY AGREE` — the plan does mention a frontend build step in Unit 8 and in the technical decisions. The gap is not "missing build step"; the gap is that it never specifies the exact command or artifact reuse strategy, and it ignores the fact that `playwright.config.ts` currently has a top-level `webServer` that will still launch Vite unless reworked.
- `C14` `DISAGREE` — `test:integration` already exists in `package.json`. The real issue is not a missing umbrella script; it is whether the new SQLite/contracts/recovery subsets intentionally roll up under the existing umbrella and how live tests are kept out of that default lane.
- `C17` `PARTIALLY AGREE` — landing boundaries would improve the plan, but the absence of commit/PR choreography is not, by itself, a blocker. It matters mainly because Unit 2 changes the published spec and becomes a dependency for later units.

## Open Points

- The plan must explicitly choose one of two scopes for Unit 5 / SC5: validate only the current OpenAPI-covered surface, or expand the OpenAPI spec first so "all API endpoints" means the full runtime API.
- The shared harness needs a concrete contract. At minimum it should specify which `OrchestratorPort` methods are stubbed by default, how seeded state enters the server, and how webhook-only deps are layered in.
- The fullstack reconnect design still needs a control mechanism. Unit 9 assumes tests can stop and restart the backend, but Unit 8 only promises `baseUrl` and `webhookSecret`.

## Additional Issues Found

- `N1` The plan does not account for Playwright's global `webServer`. Adding a `fullstack` project with "no `webServer`" does not disable the top-level Vite server in `playwright.config.ts`. This likely requires a config split, conditional config, or a separate Playwright entrypoint.
- `N2` Unit 5 introduces AJV, but `package.json` does not currently include `ajv`. The dependency change is missing from the plan.
- `N3` Unit 1's sync test can ossify the current incomplete OpenAPI surface. If the spec is missing real runtime endpoints today, a deep-equality sync test will preserve that omission unless the plan resolves scope first.
- `N4` The plan's contract-test language mixes two different targets: OpenAPI conformance and runtime route conformance. Those are not the same thing in this repo today.
