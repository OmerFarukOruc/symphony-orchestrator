# Design Context for Risoluto

## Design Context

### Product Vision

Risoluto is a **co-pilot's console**. Not a dashboard. Not a status board. Not an observability tool.

When an operator opens Risoluto, they are sitting next to an agent that is doing their engineering work, and they are watching it happen. They can intervene. They can redirect. They can wait. The interface's job is to make that sustained attention productive, honest, and beautiful â€” the way a great terminal session is beautiful to the person who lives in it.

The v2 roadmap scales this toward "Lights-Out" â€” agents dispatching agents, overnight runs, self-healing pipelines. The design language must scale from *one person watching one agent* to *one person watching a floor of agents* without changing its fundamental nature. It is always a console. It is always inhabited.

### Users

**Primary scenario:** An operator is actively managing an agent run. They are *sitting with* the interface, not checking in on it. They want to see the work as it happens, catch divergence early, and hand off cleanly when the agent finishes. The attention model is **sustained, engaged, expert** â€” not intermittent, not casual.

This is a shift from "dashboard for intermittent glances" to "console for active companionship." Design decisions should reward engagement with more information, not punish it with more chrome.

**Secondary scenarios** (must still work, but not the design target):
- Team lead reviewing overnight work the next morning â€” aggregation and exception queues matter.
- Platform engineer configuring the system â€” density and expert affordances matter.
- First-time user in the setup flow â€” progressive disclosure matters.

**User profile:** CLI-native. Docker-comfortable. Lives in terminals and IDEs. Reads dense code without complaint. Expects keyboard-forward interaction. Skeptical of warmth, friendliness, or consumer-SaaS register. Trusts interfaces that earn their attention.

**The clarity test survives the shift:** even though the primary user is an engineer watching sustained agent work, a non-developer should still be able to look at the dashboard and understand *what is happening* at a glance. Sustained attention is not the same as requiring expertise to decode. The interface still teaches itself.

### Brand Personality

Three words: **Transparent. Partnered. Alive.**

- **Transparent** â€” glass walls, not black boxes. You see exactly what the agent is doing, as it happens. Streaming output, real-time state, visible reasoning. Transparency is the product's core trust mechanism; it is not optional.

- **Partnered** â€” the console sits *next to* you, not in front of you. The interface is a co-pilot's seat, not an observation deck. It is intimate without being warm, conversational in rhythm without being chatty, present without being in the way. Think Claude Code or a great terminal session â€” the intimacy comes from the shared workspace, not from personality or friendliness.

- **Alive** â€” when agents are working, the interface is visibly inhabited. Streaming output flows. Status indicators pulse. The console *feels* occupied. This is proof of life, not decoration. When the system is idle, the interface goes still and composed. Motion is semantic in both directions: activity when there's work, stillness when there isn't.

The interface should communicate:
- "I'm working. You can see exactly what I'm doing."
- "You're in the seat next to me."
- "You can trust me with your production code because nothing is hidden."

It should not communicate:
- consumer-SaaS warmth or friendliness
- novelty-first visual choices
- decorative density without meaning
- the "another AI tool" feeling
- hostile enterprise density that dumps information without hierarchy

### Aesthetic Direction

Risoluto sits **in the overlap between two product families**:

- **Claude Code / Codex / OpenCode** â€” the terminal-native agent console family. Streaming output. Transparent activity. Conversational rhythm. Dark-first. Restrained chrome. Intimate. The partner seat.

- **Linear / Height / Raycast** â€” the premium product-engineering tool family. Surgical density. Keyboard-forward. Both themes at equal quality. Premium typography. Confident hierarchy. Sharp execution.

Neither alone is right. A pure Claude Code aesthetic would miss Linear's polish and hierarchy; a pure Linear aesthetic would miss Claude Code's intimacy and aliveness. Risoluto must feel like **a partner console built with a product engineer's craft**.

**Closest analogues:**
- **Claude Code / Codex / OpenCode** â€” the partner-console lineage. Where the intimacy comes from.
- **Linear / Height / Raycast** â€” the product-craft lineage. Where the polish and density come from.
- **Apple** â€” not the aesthetics, the *philosophy*: interfaces so clear they teach themselves. Obvious hierarchy. States that read immediately. Emotion through refinement, not decoration.

**Anti-references â€” in order of severity:**

1. **Generic AI product aesthetic** â€” cyan-on-dark, purple-to-blue gradients, glowing neon accents, glassmorphism everywhere, gradient text, rounded cards with icon + heading + text repeated endlessly. The "this was prompted from a template" look. This is the strongest do-not-become.
2. **Consumer-SaaS warmth** â€” rounded everything, friendly illustrations, chatbot avatars, emoji reactions, "hey there!" microcopy, onboarding confetti.
3. **Enterprise admin density dump** â€” endless tabs, dense tables without hierarchy, form-heavy, information dumped without interpretation. Dense â‰  good.
4. **Crypto/trading terminal intensity** â€” flashing tickers, aggressive red/green everything, Bloomberg cosplay. High signal but emotionally loud â€” wrong register for a composed partner.

### Typographic Backbone â€” Dual-Spine

Risoluto uses a **dual-spine typographic system** where monospace and proportional typography are both first-class, each owning distinct territory, with a **visibly sharp handoff** between them.

- **Proportional spine (Manrope)** â€” everything that is *the interface around the agent*: navigation, headings, labels, descriptions, body text, button chrome, empty states, tooltips, form copy. This is the operator's UI â€” calm, readable, premium.

- **Monospace spine (IBM Plex Mono)** â€” everything that is *the agent's work or its state*: log streams, structured data, identifiers (issue IDs, commit SHAs, run IDs), code, diffs, status chips, KPI numerics, table headers, activity rows. This is the agent's territory â€” the data it produces and the facts about it.

- **Display spine (Space Grotesk)** â€” headings that need to carry weight. Page titles, KPI numbers, section headers. Functions as a third voice between the two main spines â€” a heading-only layer that holds hierarchy without needing to be either mono or body.

The boundary between mono and proportional should be **visible and intentional**. When you move your eye from the nav rail (proportional) into a log row (mono), you should feel the shift. That shift is a semantic signal: *you just crossed from the UI into the agent's work*.

### Theme Philosophy

**Both light and dark are fully first-class, designed in parallel.** Light mode is not a palette-flipped dark mode and is not a fallback. It is independently hand-tuned to the same quality bar. Every component is QA'd in both. A feature that only looks good in one theme is not finished.

Risoluto cannot ship light as an afterthought, because one of its reference families (Linear, Raycast) takes light mode as seriously as dark. Shipping light as a fallback would break the engineering-product credibility the design is building.

### Design Principles

1. **Proof of life over proof of existence.** When an agent is working, the interface visibly shows it â€” streaming output, pulsing indicators, live cursors on active data. Not a spinner. Not a status dot. Real motion that reflects real work. When the system is idle, the interface goes still and composed. Motion is semantic in both directions.

2. **Dual-spine discipline.** Mono for the agent's work, proportional for the operator's UI. The boundary is sharp and intentional. Do not blur the spines to "soften" the handoff â€” the handoff is the feature.

3. **Partner, not dashboard.** Design for sustained attention, not intermittent glances. Reward engagement with information density. The operator is in the seat next to the agent; the interface should feel like *their workspace*, not *a product they're checking in on*.

4. **Glass walls, not black boxes.** Every agent action is visible and readable. The user should never wonder "what is it doing?" Transparency is the product's core trust mechanism; it is not optional.

5. **Self-teaching clarity.** Even though the primary user is a sustained-attention expert, a non-developer should still be able to look at the dashboard and understand *what is happening* at a glance. Sustained attention â‰  requiring a manual. The interface always teaches itself.

6. **Semantics over decoration.** Copper for brand and primary action only. Status colors for runtime meaning only. No color used decoratively. Every visual signal earns its presence.

7. **Both themes, equal quality.** Light and dark both designed to the same standard. A feature that only looks good in one is not finished.

### Dual-Spine Rules

The dual-spine system is an explicit design choice â€” mono and proportional each own territory, and the boundary between them should be visible.

**Proportional spine â€” use `--font-body`/`--font-sans`:**
- Navigation, sidebars, breadcrumbs
- Page titles and section headings (via `--font-heading` for display sizes)
- Body text, descriptions, help copy
- Button chrome, form labels, empty-state messaging
- Tooltips and overlay text

**Monospace spine â€” use `--font-mono`:**
- Activity logs, streaming output, terminal-style panels
- Data tables, status chips, badges, chip identifiers
- Issue IDs, commit SHAs, run IDs, thread/turn IDs
- Diffs and code
- KPI numerics (with `--font-features-tabular`)
- Table column headers (uppercase, mono, tracking-wider)
- Keyboard hint pills (`.mc-button-hint`)

**Display spine â€” use `--font-heading`:**
- `.page-title` and `.heading-display` (hero numbers, KPI values)
- `.heading-section`, `.heading-card`

**The boundary is the feature.** Do not mix mono and proportional within a single line unless you're intentionally marking a shift (e.g., proportional label + mono value). Do not soften the boundary with a third font. Do not substitute one spine for the other to "match" a neighboring element â€” let them stay in their own territories.

## Surface Hierarchy

The live system formalizes a three-tier surface taxonomy. Treat this as the authoritative way to reason about surface weight in any new UI work.

### Primary Surface

**Use for:** Command panels, live data displays, primary actions, active run panels, command input areas, live status cards.

**Character:** Commanding presence, highest visual weight. The eye should find these first. These are where the agent's *current* work lives.

**Features:** Copper accent border, subtle elevation shadow, slightly elevated background, stronger default border.

**Tokens:** `--surface-primary-border`, `--surface-primary-border-accent`, `--surface-primary-bg`, `--surface-primary-border-width`, `--surface-primary-accent-width` (3px), `--surface-primary-shadow`, `--surface-primary-text`.

### Standard Surface

**Use for:** Default containers, list items, content cards, issue cards, event rows, stat cards, toolbars.

**Character:** Balanced, readable, default treatment.

**Features:** Basic 1px border, surface background, no accent.

**Tokens:** `--surface-standard-border`, `--surface-standard-bg`, `--surface-standard-border-width`, `--surface-standard-shadow`, `--surface-standard-text`.

### Quiet Surface

**Use for:** Metadata, secondary info, supporting content, empty states.

**Character:** Minimal presence, recedes visually. Should never compete for attention.

**Features:** No border, muted background, lowest contrast.

**Tokens:** `--surface-quiet-border` (transparent), `--surface-quiet-bg`, `--surface-quiet-shadow`, `--surface-quiet-text`.

### Hierarchy Rules

1. Primary surfaces should draw the eye first
2. Standard surfaces provide the main content structure
3. Quiet surfaces should never compete for attention
4. **Never nest same-tier surfaces** â€” no primary within primary, no standard within standard

### Running-State Live Indicator â€” The Core Proof-of-Life Affordance

The `.is-status-running::before` pseudo-element implements a GPU-optimized glow pulse (`statusGlowGpu` keyframe, 2s ease-in-out, opacity 0.5 â†’ 1). This is the canonical "agent is alive" affordance and is load-bearing for the "proof of life" design principle.

When adding new components that represent running or active state, **extend this pattern rather than inventing a new one**. The consistency is the feature: users learn that the pulse means "live" once, and then recognize it everywhere.

### Known Pattern Exception â€” Toast Border-Left

`primitives.css` defines `.toast-success`, `.toast-error`, `.toast-warning`, `.toast-info` using `border-left: 3px solid`. This is the single living instance of the "side-stripe border on alert" pattern that the `/impeccable` skill's `absolute_bans` explicitly forbids. It is kept because toasts are transient, high-signal, and the stripe carries semantic meaning (status color) in under 200ms of on-screen time.

**Do not extend this pattern to persistent surfaces** (cards, list items, callouts, panels). Those must use alternative indicators (status dots, background tint, or container state classes).

## Implementation Guidance

When designing or refactoring UI in this project:

1. **Use `mc-*` primitives and tokens** over page-local one-offs. If a pattern is missing, propose adding it to `components.css` rather than inventing a local class.
2. **Respect the dual-spine.** Mono for the agent's work (logs, data, IDs, code). Proportional for the operator's UI (nav, headings, body, labels). The handoff should be visible â€” don't blur it.
3. **Use semantic status colors for runtime state only.** Never copper for status. Never status colors for decoration. Every color earns its presence.
4. **Copper for brand, active navigation, and primary action only.** One copper accent per viewport is usually enough.
5. **Preserve the zero-radius stitch.** Major surfaces stay square. Only compact controls may soften.
6. **Every change must work in both light and dark themes.** Verify explicitly â€” a feature that only looks right in one is not finished.
7. **Respect the surface hierarchy.** Pick Primary, Standard, or Quiet deliberately. Never nest same-tier surfaces.
8. **Hit WCAG AA.** 4.5:1 for body text, 3:1 for large text and UI components. The text-* tokens are already tuned.
9. **Use the two-layer type scale correctly.** For product UI, reach for `--text-ui`, `--text-sm`, `--text-sm-plus`, `--text-md` (app layer). For marketing or large displays, use `--text-2xl`, `--text-hero` (fluid).
10. **Apply tabular numerals to anything numeric that stacks or aligns** (`font-feature-settings: var(--font-features-tabular)`).
11. **Motion is proof of life.** Running = visibly alive (pulse, streaming, progress). Idle = still and composed. No decorative animation. Always honor `prefers-reduced-motion`.
12. **Extend the status-running pulse** for any new live-state component. The consistency is the feature.
13. **Every new feature must be transparent:** agent activity, state transitions, and cost must be readable without prior knowledge.
14. **Self-teaching test still applies:** can a non-developer understand what's happening at a glance? If not, the design is not done.
15. **Invoke `/visual-verify`** after editing any CSS, `dashboard-template.ts`, or `logs-template.ts`. Mandatory before marking UI tasks done (see `CLAUDE.md`).
16. **Update this file when the live frontend changes materially**, not before. This document trails the code; it does not lead it.
