# Design Context for Risoluto

## Design Context

### Product Vision

Risoluto is not a dashboard tool. It is an **autonomous engineering operating system**. The v2 roadmap ends at Phase 7 — "Lights-Out": agents dispatching agents, self-healing pipelines, multi-agent role pipelines running overnight, auto-merging PRs without a human in the loop.

The interface must be designed with that trajectory in mind. Today it shows one person their agents. Tomorrow it shows a team their autonomous engineering floor. The design language must scale credibly to both.

### Users

**Primary:** Software engineers and solo operators who run Risoluto locally to manage Linear/GitHub issue lifecycles through sandboxed AI agents. CLI-native, Docker-comfortable, used to tool density. They open the dashboard intermittently — to check in, not to sit and operate. They need to understand what's happening in seconds without a tutorial.

**Design test:** Could a non-developer — a product manager, a founder, a curious family member — look at the dashboard and understand what's happening? Not configure it, but *understand* it. If the answer is no, the design has failed. This is the Apple standard: the interface teaches itself through clarity, not through documentation.

**Attention model:** Intermittent, not continuous. Users glance, scan, act on exceptions, leave. The interface must be self-reading at a glance and never require re-learning.

### Brand Personality

Three words: **Transparent. Calm. Inevitable.**

- **Transparent** — like Codex, Claude Code, and OpenCode: you see everything the agent is doing, as it happens, without cognitive overhead. No black boxes. Glass walls, not locked rooms.
- **Calm** — not excited, not urgent, not playful. Composed under pressure. The product runs overnight; it should *feel* like it can be trusted to do that.
- **Inevitable** — as the system scales toward lights-out autonomy, the design should feel like precision engineering: exactly what it needs to be, nothing extra.

The interface should communicate:
- "I know what I'm doing."
- "You can trust me with your production code."
- "Nothing is hidden from you."

It should not communicate:
- consumer-SaaS warmth or friendliness
- novelty-first visual choices
- decorative density without meaning
- the "another AI tool" feeling

### Aesthetic Direction

**Visual reference:** The feeling of watching a senior engineer work in an open-plan office — you can see their screen, their terminal, their progress. It's readable without being noisy. It respects your attention.

**Closest analogues:**
- **Apple** — not the aesthetics, the *philosophy*: interfaces so clear they need no manual. Hierarchy that's obvious. States that read immediately. Emotion through refinement, not decoration.
- **Codex / Claude Code / OpenCode** — transparent agent activity, streaming output, real-time visibility into what the agent is producing. No mystery. The work is visible.
- **Linear** — surgical information density, fast, no fluff, confident typographic hierarchy, both themes treated with equal seriousness.

**Anti-references:**
- Cyan-on-dark "AI product" gradient aesthetic
- Purple-to-blue gradient backgrounds
- Generic Tailwind card grids
- Glassmorphism used decoratively
- Bubbly rounded cards with icon + heading + text repeated endlessly
- Anything that looks like it was prompted from a template

**Color system:** Copper is the brand and primary-action accent. Semantic status colors are operational — they carry meaning, not decoration. Both themes share this system with equal intentionality.

**Shape language:** Zero-radius on major surfaces (the "stitch" aesthetic). Small radii acceptable on compact controls. Crisp, composed, stitched — not pillowy.

**Motion:** Motion should reflect real state. When agents are running, the interface should feel alive — subtle pulse, live activity. When idle, stillness and confidence. Never decorative animation.

### Theme Philosophy

**Both light and dark are first-class targets.** Light mode is not a color-inverted dark mode and is not a fallback. It is a fully considered experience, designed for operators who work in bright environments or who simply prefer light UIs. The same design principles — copper accent, semantic status colors, zero-radius stitch, transparent agent activity — apply with equal fidelity in both.

A design that only looks good in dark mode has not been fully designed.

### Design Principles

1. **Glass walls, not black boxes** — Every agent action is visible and readable. The user should never wonder "what is it doing?" Transparency is not optional; it is the product's core trust mechanism.

2. **Self-teaching clarity** — If a non-developer can't understand the current state of the system at a glance, the design has failed. No tooltips required. No documentation required. The interface reads itself.

3. **Semantics over decoration** — Copper for brand and primary action only. Status colors for runtime meaning only. No color used decoratively. Every visual signal earns its presence.

4. **Earned density** — Information shown progressively, not dumped. Scan the overview; drill into the detail. The dashboard teaches you what matters most before showing you everything.

5. **Both themes, equal quality** — Light and dark are both designed to the same standard. Neither is an afterthought. A feature that only looks good in one theme is not finished.

### Radius

- `--radius-sm`: 2px
- `--radius-md`: 2px
- `--radius-lg`: 0px
- `--radius-xl`: 0px
- `--radius-full`: 9999px

Compact controls can have slight softening. Major surfaces stay visually square.

### Controls And Motion

- Control heights: `--control-height-xs` (28px) through `--control-height-xl` (44px)
- Motion timing: `--motion-instant`, `--motion-fast`, `--motion-medium`, `--motion-slow`
- Motion reflects real state: agents running = visible activity; idle = stillness
- No ornamental animations, no bounce, no decorative transitions
- Full `prefers-reduced-motion` support required

## Implementation Guidance

When designing or refactoring UI in this project:

1. Use `mc-*` primitives and tokens over page-local one-offs.
2. Use semantic status colors for runtime state — never copper.
3. Use copper for brand, active navigation, and primary action only.
4. Preserve the sharp, composed, information-dense visual language.
5. Always verify token usage works in both light and dark themes.
6. Every new feature must be transparent: agent activity, state transitions, and cost must be readable without prior knowledge.
7. Apply the self-teaching clarity test: can a non-developer understand what's happening at a glance?
8. Update this file when the live frontend changes materially, not before.
